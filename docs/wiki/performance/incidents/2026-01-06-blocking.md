## Root Cause Analysis

### Primary Cause
A bulk operation (likely data import or batch update) triggered `trig_DetailUpdate` on many rows simultaneously.

### Cascading Effect
```
Bulk INSERT/UPDATE to Detail table
    │
    ▼
trig_DetailUpdate fires
    │
    ├── Holds locks on Detail (X locks)
    ├── Holds locks on Items (S locks)
    ├── Holds locks on CrossRefs (S locks)
    │
    ▼
Other queries blocked:
    ├── CrossRef lookups (737 min blocked)
    ├── uf_RequisitionIsVisible (63 min blocked)
    ├── DELETE FROM vw_ReqDetail (31 min blocked)
    └── SELECT from vw_ExistingRequisitions (27 min blocked)
```

### Contributing Factors
1. **Trigger complexity** - `trig_DetailUpdate` performs too many operations
2. **Lock escalation** - Row locks escalated to page/table locks
3. **No batch size limit** - Bulk operation processed all rows at once
4. **Peak timing** - Occurred during 22:00 batch job window

## Prevention Measures

### Implemented
- [ ] Pending implementation

### Recommended
1. **Batch processing limits**
   - Process maximum 1000 rows per transaction
   - Add delay between batches

2. **Trigger optimization**
   - See [KI-003: trig_DetailUpdate](../known-issues/trigger-blocking.md)
   - Add NOLOCK hints for read operations
   - Move non-critical logic to async

3. **Monitoring enhancements**
   - Alert when blocking > 10 minutes
   - Alert when deadlocks occur
   - Dashboard for real-time visibility

4. **Scheduling changes**
   - Avoid bulk operations during business hours
   - Stagger batch jobs to avoid overlap

## Lessons Learned

| Lesson | Action |
|--------|--------|
| Triggers can cause cascading blocking | Review and optimize all triggers |
| Bulk operations need limits | Implement batch size limits |
| 22:00 is a risky time | Review job scheduling |
| Need better monitoring | Implement proactive alerts |
